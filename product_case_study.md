### 关键模块设计与疑问解答：

1.  **数据服务 (Data Service) - ETL 引擎**：
    *   **定义**：这是架构图中的 "基础设施层"。它不是一个单一的脚本，而是一套 **ETL 管道 (Extract-Transform-Load)**。
    *   **核心职能**：
        1.  **标准化 (Normalization)**：将 OKX 的 API JSON 数据和 RSS 新闻数据清洗为统一的 Pandas DataFrame (CSV) 格式。
        2.  **特征工程 (Feature Engineering)**：代码 `generate_multi_coin_signals.py` 在这里工作。它计算 20+ 种技术指标（如 RSI, MACD, 资金费率 Z-Score）。
        3.  **数据对齐 (Alignment)**：将毫秒级的新闻时间戳与 4小时级别的 K 线对齐，确保 AI 训练时不会产生 "未来函数"。
    *   *面试话术*："AI 的质量取决于数据的质量。我构建了一个标准化的 ETL 中间层，确保无论是来自交易所的结构化数据，还是来自 Twitter 的非结构化数据，在进入 AI 模型前都经过了清洗和对齐。"

2.  **新闻语境处理器 (News Context Processor)**：
    *   **定义**：这一模块 (`get_news_context`) 充当了全职的 **"宏观研究员"**。它在数据进入大模型前进行 **预处理 (Preprocessing)** 和 **降噪**。
    *   **核心职能**：
        1.  **宏观日历过滤**：提取 Economic Calendar，锁定本周如 CPI/非农 等关键事件。
        2.  **爆仓数据翻译**：将离散的 Liquidation 数字翻译为语义逻辑（如：“多头爆仓量是空头的2倍 -> 提示可能存在超卖反弹”）。
        3.  **美联储预期**：计算 Fed Funds Futures 的隐含利率，判断市场是否处于“降息预期交易”中。
    *   *面试话术*："为了防止 LLM 被噪音淹没，我设计了这个中间层。它不只是转发新闻，而是生成一份结构化的 **Daily Briefing (每日简报)**，确保 DeepSeek 能够基于高质量的宏观语境做决策。"

3.  **架构的一致性 (Paper Trade vs API)**：
    *   **架构逻辑**：在这个架构中，**订单管理系统 (OMS)** 与 **执行层** 是解耦的。OMS 发出标准的 "买入/卖出" 指令，底层由 **执行适配器 (Adapter)** 决定是发送给真实的 OKX API 还是 本地的模拟撮合引擎 (Paper Trade)。
    *   **面试话术*："我们在 MVP 阶段使用 Paper Trade 模块来零风险验证策略，但在架构设计上，我们预留了标准接口，只需切换 Config 即可无缝迁移到实盘。" —— 这正体现了 **高内聚、低耦合** 的系统设计能力。

4.  **订单管理系统 (OMS) 工作流详解**：
    *   **定义**：OMS 是连接 "大脑 (Agent)" 和 "手 (Exchange)" 的中枢神经。
    *   **核心职能拆解**：
        1.  **指令标准化 (Standardization)**：Agent 输出的是 "意图" (如：*我要买 $1000 BTC*)。OMS 将其翻译为交易所能懂的 "订单" (如：*Type=Market, Side=Buy, Size=0.012 BTC*)。
        2.  **生命周期管理 (Lifecycle)**：
            *   **New**: 开新仓。
            *   **Amend**: 加仓/减仓 (Merge Logic)，或者调整止损 (Adjust SL)。
            *   **Cancel/Close**: 平仓。
        3.  **资金校验 (Pre-Trade Validation)**：在发送给交易所前，OMS 会再次核对本地账本 (Cash)，确保 "余额充足"，避免因交易所 API 报错导致系统崩溃。
    *   *面试话术*："我设计的 OMS 具备 **状态机 (State Machine)** 的特性，能够处理复杂的订单生命周期，比如'加仓合并'和'动态调整止损'，而不仅仅是简单的发单。"

5.  **DeepSeek 的决策依赖 (Log vs State)**：
    *   **不需要看"全量日志"**：DeepSeek 每次决策主要依赖 **"当前状态" (Portfolio State)**，即当前持仓是多少、浮盈浮亏是多少。
    *   **需要看"上一轮决策"**：它需要知道上一轮对这个持仓的 *规划* (如止损位)。
    *   *流程*：系统会将 `portfolio_state.json` (持仓状态) 喂给 DeepSeek，而不是让它去翻阅历史文本日志。这被称为 **"Stateless Execution with Context" (带状态上下文的无状态执行)**，效率更高。

5.  **反馈闭环的数据流 (Data Flow Details)**：
    *   **From 执行适配器 (Execution Adapter)**：
        *   **返回数据**：`portfolio_state.json` (持仓快照)。
        *   **包含字段**：`positions` (当前持仓的入场价、数量), `nav` (净值), `cash` (可用资金)。
        *   **作用**：告诉 Agent "你的上一笔订单是否成交？现在的真实仓位是多少？" —— 只有知道自己持有 BTC，Agent 下一轮才会考虑 "Reduce Position" 或 "Take Profit"。

6.  **核心模块设计**：
    1.  **推理引擎 (核心产品)**：
        *   **Quant (量化)**：使用 Microsoft Qlib (LightGBM) 生成基础评分 (Base Rate)。
        *   **Qual (定性)**：使用 DeepSeek LLM 进行 "语境感知" (Narrative Check)。
        *   *PM 洞察*："我们将'逻辑推理'与'数学计算'解耦，防止大模型在处理数值时产生幻觉。"

    2.  **中台系统 (风险引擎)**：
        *   **核心逻辑**：采用 **"三道防线 (Three-Layer Defense)"** 的漏斗式设计，充当交易的最终守门员（Gatekeeper）。
        *   **第一道防线：微观风控 (Volatility Guard)**
            *   当检测到市场波动率 (ATR) 为 "High" 时，强制将所有新开仓位的 **杠杆降为 1x**。
            *   *目的*：防止在插针行情中被瞬时爆仓。
        *   **第二道防线：宏观风控 (Dynamic Exposure Cap)**
            *   **牛市模式** (BTC > SMA50 且 恐慌指数 > 40)：允许 100% 仓位敞口。
            *   **熊市/防御模式**：强制限制总敞口为 **NAV 的 50%**。
            *   *目的*：自动化管理 Beta 风险，在下跌趋势中强制“保住本金”。
        *   **第三道防线：集中度风控 (Concentration Limit)**
            *   严格限制同时持仓不超过 **3 个标的**。
            *   *目的*：防止 AI "分散撒网"，确保每一笔交易都是高确信度 (High Conviction) 的精品。

## 3. "跟单交易"的产品逻辑
*如何将其包装为一个面向用户的商业产品。*

**痛点**：散户希望获得 "Smart Money" 的收益，但传统跟单交易常导致 "收割" 或 "爆仓"，因为带单员（Master Trader）风格激进且不透明。
**解决方案**：**"透明化 AI 策略 (Transparent AI Strategies)"**。
*   **差异化**：不同于隐藏逻辑的人类交易员，Dolores 实时发布 **决策日志 (Decision Log)**（如 `agent_decision_log.json`），解释 *为什麽* 进行这笔交易（例如："检测到资金费率轧空风险"）。
*   **功能亮点**："推理证明 (Proof of Reasoning)"。用户不仅仅是复制 *交易*，更是订阅 *投资逻辑*。

## 5. 我的核心优化 (Vs. 竞品/基础版 nofi.ai)
*这是你的加分项：展示你如何在一个基础产品上通过"产品思维"使其更好。*

我在基础的 AI 炒股模式（如 nofi.ai 早期版本）之上，进行了 **4 维度的深度优化**：

### 1. 逻辑层：引入"反直觉"的二阶思维 (Second-Order Thinking)
*   **基础版痛点**：传统 AI 看到"好新闻"就买入，容易成为"接盘侠"。
*   **我的优化**：增加了 **"叙事校验 (Narrative Check)"** 模块。
    *   *逻辑*：如果 `新闻是利好` 且 `价格未涨` = **利好出尽 (Priced In)** -> **看跌信号**。
    *   *价值*：使 Agent 具备了类似对冲基金经理的"反向博弈"能力。

### 2. 信号层：引入"市场微观结构" (Market Microstructure)
*   **基础版痛点**：仅依赖 K 线指标（RSI/MACD），容易被庄家画线诱多/诱空。
*   **我的优化**：增加了 **"痛苦交易检测 (Pain Trade Detection)"**。
    *   *数据源*：监控 **资金费率 (Funding Rate)** 和 **清算数据 (Liquidations)**。
    *   *逻辑*：识别"拥挤交易"。如果 `所有人都在做多 (费率极高)` = **轧多风险 (Long Squeeze)** -> **拒绝买入**。

### 3. 风控层：从"静态止损"升级为"动态防御"
*   **基础版痛点**：只有简单的固定止损，缺乏宏观风控。
*   **我的优化**：构建了 **"中台风控引擎"**。
    *   **波动率熔断**：高波动环境下强制降杠杆至 1x。
    *   **熊市保护**：当 `价格 < SMA50` 时，强制将总仓位敞口限制在 NBA 的 50% 以内，优先保本。

### 4. 成本与模型：DeepSeek-V3 降本增效
*   **基础版痛点**：使用 GPT-4 API 成本高昂，推理速度慢。
*   **我的优化**：迁移至 **DeepSeek-V3**。
    *   *效果*：API 成本降低 90%，且在数理逻辑推理（Reasoning）上的表现更适合金融场景，实现了"高频思考，低成本决策"。

## 7. 深度对比：从 Nofi.ai (v1) 到 Dolores (v2) 的进化
*基于原始提示词的分析，展示你的架构升级优势。*

通过分析 nofi.ai 的原始提示词（主要是一个详细的文本行研报告），可以看出你的 **Dolores** 项目实现了从 **"辅助分析工具"** 到 **"全自动交易系统"** 的跨越。

| 维度 | Nofi.ai (原始模式) | Dolores (你的优化版) | 核心优势 (你的卖点) |
| :--- | :--- | :--- | :--- |
| **数据输入 (Input)** | **非结构化文本**：依赖大段的文字描述（如 "Rel. Vol: Elevated"），模糊且难以回测。 | **结构化因子 (Quant)**：直接输入 Qlib 的数值评分 (Rank #1, Score 0.68) 和 Z-Score。 | **精确性**：将"感觉"转化为"数学"，消除了语言的模糊性。 |
| **决策逻辑 (Logic)** | **纯大模型 (Pure LLM)**：试图让 LLM 理解所有对冲基金黑话（"Liquidity Trap", "Pain Trade"），依赖 LLM 的"文科"理解力。 | **神经符号架构 (Neuro-Symbolic)**：**Quant (数学)** 负责胜率，**LLM (推理)** 负责赔率和宏观。将"计算"剥离给 LightGBM/Pandas，让 LLM 专注"逻辑"。 | **更抗幻觉**：LLM 不擅长算术，但 Qlib 擅长。两者结合优于单一大模型。 |
| **信号来源** | **文本化数据**：如输入 "Funding: -0.01%"。LLM 难以感知这个数字在历史上的分位点。 | **统计学特征**：直接输入 **Z-Score** (标准分)。告诉 LLM "这是 3 倍标准差的异常值"，而非仅给裸数据。 | **感知力强**：让 AI 理解数据的"统计显著性"。 |
| **风险控制** | **模型内置风控**：依赖 LLM 也就是 Prompt 里的 "Confidence" 字段自我约束。 | **中台硬风控**：代码写死的规则（SMA50 熊市线）。即使 LLM 想买，代码也会在最后一步拦截。 | **确定性**：代码约束 > 概率约束。 |

**总结你的 "Alpha" (超额收益来源)：**
Nofi.ai 的原始模式更像是一个 **"超级助理"**，帮交易员整理信息；而 Dolores 是一个 **"合成交易员"**，它拥有：
1.  **量化的眼睛 (Qlib)**：看清数据。
2.  **AI 的大脑 (DeepSeek)**：读懂新闻。
3.  **机器的手 (Risk Engine)**：严格执行。
